## 2. Frontend invariants INV-FE-1…15 — normative texts (F-3)

The superseded FRONTEND_PLAN.md cited INV-FE-1…15 at ~79 sites and certified against them (§32) without ever stating their texts (review F-3). The texts below are published as normative. **From this document forward, frontend certification (§6) binds to these texts and to nothing else.** Provenance of each text is declared in §2.2; where the source usage was too thin to pin a text exactly, the text below is the most conservative statement consistent with every citation site.

> **INV-FE-1 (authoritative reads).** The client-embedded light client is the only authoritative read path. Every transaction-critical value — any value whose incorrectness could change what a user signs or their understanding of what signing does — is ultimately obtained from **finalized** chain state whose proofs the light client verified, or computed client-side purely from such verified inputs. No RPC endpoint, indexer, provider, cache, or best-head datum may be the source of a transaction-critical value. Data obtained over the quarantined RPC fallback is **never promoted** to verified status; verified status requires a light-client re-read (D-6/F-2, [10-frontend-architecture.md](10-frontend-architecture.md)).
>
> **INV-FE-2 (pre-sign revalidation).** Immediately before signature, every declared precondition of the transaction is re-read at a single finalized block, and the live runtime version is re-checked against the version the payload was built for; any failure blocks signing with a diff of expected vs actual. This refresh is structural on every submit path — no code path can reach a signer without it.
>
> **INV-FE-3 (acceleration firewall).** Optional acceleration (community indexers, snapshots, any historical provider) is never transaction-critical: provider-supplied data never satisfies a precondition, never renders a protocol-outcome state ("passed / settled / mature / final / safe") without a direct chain read, and is barred from the transaction path **structurally** — by build-time package dependency boundaries enforced in CI, including inside the application package itself — not by reviewer vigilance.
>
> **INV-FE-4 (zero-infrastructure operation).** Every protocol workflow — epoch/proposal discovery; proposal detail including reading the exact committed proposal payload bytes and hash-checking them client-side; trading; positions, split/merge/transfer and every redemption path including `Voided`; proposal submission; oracle reporting and challenges; execution-queue interaction and permissionless cranks; governance referenda (vote/delegate/unlock, ratification, `OracleResolution`); operator workflows; and USDC funding/withdrawal ([11-frontend-workflows.md](11-frontend-workflows.md)) — operates against finalized, light-client-verified chain state with **zero optional infrastructure available**: no indexer, no RPC endpoint, no provider, cleared local storage.
>
> **INV-FE-5 (no key custody).** The application never holds, derives, stores, or transmits private keys or any signing secret. Signing occurs exclusively in the user's chosen wallet (browser extension or raw-payload external signer); the application boundary carries only encoded payloads outward and signatures inward.
>
> **INV-FE-6 (serverless client).** The canonical frontend is a static, content-addressed bundle: no application backend, no server-side rendering, no required RPC endpoint, no required indexer, and no feature whose correctness depends on any server operated for the application. Features that inherently require servers are out of scope rather than centralized.
>
> **INV-FE-7 (disposable local storage).** Browser-local storage is a non-authoritative cache and derived-history store. Its loss, corruption, or eviction is a performance and convenience event only: the transaction path never reads it, protocol workflows are unaffected, and the application rebuilds it automatically from chain data.
>
> **INV-FE-8 (single-operator tamper evidence).** No single operator — Arweave gateway, ArNS controller, RPC host, chain peer, indexer, snapshot publisher, or the CI system — can silently alter the application or its interpretation of chain state. Every such channel carries a detection mechanism: content-hash verification and the signed release self-check (distribution), light-client proof verification and quote double-derivation (chain), sampling and snapshot diffing (acceleration), multi-gateway resolution cross-checks (naming). Detected divergence is surfaced to the user; it is never silently repaired.
>
> **INV-FE-9 (provenance labeling).** Every displayed data item carries an explicit, typed verification status (`verified-finalized`, `verified-best`, `derived-local`, `provider`, `stale-cache`). Unfinalized/best-head data is display-only and never transaction-critical. There is no unlabeled rendering path: UI data components reject unlabeled values by type.
>
> **INV-FE-10 (reproducible releases).** Every release is reproducibly buildable: independent parties following the published recipe obtain byte-identical output whose tree hash equals the published, multi-party-signed release manifest; release identity is content-addressed. A user can independently verify *which* application they run, from source commit to served bytes, without trusting the release team or CI.
>
> **INV-FE-11 (pinned identity, visible provenance).** The bundle pins the complete identity of what it is and what it talks to — release content address (TXID), source commit, per-file hashes, descriptor metadata hashes, supported spec-version range, chain-spec hashes, and relay + parachain genesis hashes — verifies the chain identity at boot (genesis mismatch is terminal), and displays all of it in an always-available verification panel.
>
> **INV-FE-12 (fail-safe under unknown runtimes).** When the runtime surface is unknown or partially incompatible, the application fails safe into explicit `restricted` or `read-only-incompatible` modes: signing is disabled wherever compatibility is unproven, reads continue only where compatibility probes pass, undecodable data renders as raw SCALE with an explicit warning, and the application **never guesses at encodings**.
>
> **INV-FE-13 (no telemetry, no remote configuration).** The application emits no telemetry or analytics of any kind, and has no remote-configuration channel: every provider suggestion, locale, bootnode list, default and text ships inside the signed release; user settings live only on the device. Application behavior changes only by shipping a new, independently verifiable release.
>
> **INV-FE-14 (signing transparency).** Everything a user signs is fully inspectable before signing: a human-readable summary, a structured decoded tree, and the exact raw SCALE bytes with their payload hash — all derived from the same bytes that will be signed, never from UI form state — plus signed-extension details. Expert mode exposes the raw storage keys and SCALE values behind every precondition.
>
> **INV-FE-15 (replaceable, reproducible, labeled acceleration).** Historical acceleration is optional, user-selected, and interchangeable: snapshots are deterministic, content-addressed exports that anyone can reproduce and verify byte-for-byte; live providers implement a minimal open read-only interface anyone can operate; every provider-derived row carries its origin to the pixel; providers are sampled against chain state where feasible and auto-disabled on mismatch. Historical analytics **degrade gracefully**: verified data is shown as verified; everything else is either absent with an explanation or present and labeled — gaps are first-class and visible, never silently spliced (D-6).

### 4.5 Fuzzing, benchmarks, weights

cargo-fuzz on SCALE decoding of payload batches, nested-wrapper filtering (incl. `proxy_announced`, `as_multi_threshold_1`), and LMSR trade paths. `frame-benchmarking` for every call and hook; CI gates on > 10 % weight regressions; PoV-size tracking against the budgets derived in [13-parameters.md](13-parameters.md) (196-market table). **Supply-chain gates (TH-34):** every backend CI run asserts the committed lockfile (`cargo --locked`) and a clean advisory pass under the [01 §9](01-system-overview.md) exact-pin regime; the §5 release pipeline MUST NOT publish artifacts from a run with either gate red.

**Advisory sources (normative; amended 2026-07-20, SQ-219).** The normative advisory sources are **RustSec *and* the GitHub Advisory Database (via OSV)**. RustSec alone is insufficient and the insufficiency is structural, not operational: for crates.io the GitHub Advisory Database is a strict *superset*, so an advisory may exist with no RUSTSEC id at all, against which `cargo-audit` is blind rather than misconfigured. The two legs MUST partition the space rather than overlap — `cargo-audit` gates everything RustSec covers (preserving its `unmaintained`/`unsound` informational classification and its annotated exception file), and a pinned OSV leg gates **exactly the complement**: a finding is treated as RustSec-covered only when it carries a RUSTSEC id itself, or an alias that is itself a RustSec record *for that same package* — an alias pointing at a RustSec record for a sibling package confers no coverage. Waivers on the complement leg are per-entry annotated, keyed to `(id, package, version)` so a version bump cannot silently inherit one, and **expire mechanically**: a waiver matching no live finding fails the gate.

**The three fuzz areas are exhaustive by construction (normative; amended 2026-07-20, SQ-226/SQ-227).** The areas named above are the complete `cargo-fuzz` obligation; an invariant not named here is verified by its own method in §1's Verification column, not by an unstated fuzz duty. Fuzz targets MAY be built against the **frame-free functional cores** rather than the aggregate runtime types: the fuzz workspace is deliberately isolated (its own nightly toolchain, outside the runtime workspace's exact-pin closure) so that libFuzzer and nightly can never perturb the pinned SDK line, and linking the aggregate `RuntimeCall` would dissolve that isolation. Where a target fuzzes a core type **whose production counterpart has a different decode or dispatch shape** — so that the fuzzed type is a *model* of the production boundary rather than the boundary itself — that production boundary MUST carry its own hardening plus a named regression test in the runtime suite; for payload batches that is the SCALE decode-depth limit and its fail-closed regression (SQ-225). Model-type fuzzing without that runtime-side pairing is **not** a sufficient discharge. Where the core *is* the production path the pallet delegates to (LMSR trade paths), no such pairing arises and none is required.

**Weights and the regression gate (normative; amended 2026-07-20, SQ-147/SQ-148/SQ-149).**

- The > 10 % gate is discharged by a deterministic **diff of the committed generated weight artifacts** against the merge base — worst-case totals (ref-time, proof-size, reads, writes) evaluated at each revision's component high bounds, over the generated weight files the runtime actually dispatches with. Three qualifications are deliberate and normative: a regression whose *absolute* delta falls below a per-dimension noise floor does not trip the gate; an intentional regression passes only under a committed, justified acknowledgement that **expires mechanically** (a stale acknowledgement fails the gate, as does an unacknowledged removal); and CI does **not** force benchmark regeneration when benchmarked source changes — the per-commit benchmark job proves executability, not calibration, so regeneration is an authoring and release-time duty. Because the gate measures *growth*, it cannot see absolute headroom: the complementary standing obligation is that every call and hook, at its worst component arguments, fits its dispatch class, with the [13](13-parameters.md) §5 per-call PoV estimates pinned as regression ceilings.
- **Charged weight of a payload-executing extrinsic.** [09](09-execution-upgrades-and-rollout.md) §1.2(11) bounds a batch's *declared* weight; it does not fix what the executing extrinsic charges. Such an extrinsic MUST **pre-charge** its own overhead at the call-count bound plus the full declared payload ceiling, and refund actuals via post-dispatch info. Where the dispatcher does not surface inner post-dispatch weight, the refund MUST fall back to the **declared** inner weights — never to a value that could undercharge. The no-undercharge property rests on the pre-dispatch check that the declared sum is within the payload ceiling, not on any debug-only assertion.
- **Batched cranks are weighed worst-case heterogeneous.** A batch extrinsic is benchmarked and charged as if **every** item carries a distinct, maximal payload; a shared or undersized fixture is forbidden, since it collapses the per-item proof slope (the shared-preimage fixture that undercharged PoV ≈ 4× is the standing justification).

**Waiver discipline (normative; added 2026-07-20).** The exact-pin regime and a clean advisory gate are in genuine tension: an advisory can be un-actionable inside the pin regime because the vulnerable version is forced by a pinned SDK component, and unpinning to clear it would breach [01 §9](01-system-overview.md). "Clean" therefore means *clean under a declared, annotated exception set*, subject to all of:

1. **Per-entry annotation.** Every waived advisory names the component that forces the vulnerable version and a **per-family exit criterion** — the upstream condition that clears it. A waiver with no exit criterion is not admissible.
2. **Pin-forced proof.** Each entry is admissible only while the pin genuinely forces it; an attempted `cargo update --precise` (or equivalent) MUST be recorded with its blocker. An advisory that is merely inconvenient to fix is not waivable.
3. **No stale waivers.** A waiver MUST NOT outlive the finding that forced it: the advisory-complement leg fails on a waiver matching no finding, and the exception set as a whole is **re-triaged from scratch at every SDK train bump** — never carried forward unexamined — so an entry whose pin has moved cannot survive one.
4. **Blast-radius containment.** A waiver is scoped to the workspace whose pins force it. A separate workspace outside the pinned SDK closure — the keeper workspace, [01 §9](01-system-overview.md) — is audited from its own root, so no SDK-forced exception can mask a vulnerability there.
5. **Disclosure.** Every **publishable** release manifest (§5) carries the full waived-advisory list; a release whose manifest omits it is not publishable.

Waiver *count* is never a gate — a small list of well-understood pin-forced entries is healthier than a green gate bought by silence. This clause governs the waiver mechanism only; which advisory databases are normative sources for the gate is a separate question.

### 4.9 Economic simulation, contests, audits

Agent-based adversarial simulation (informed traders with planted ground truth, noise traders, arbitrage capital, scripted manipulators with [14](14-threat-model.md) budgets) across ≥ 10⁴ synthetic proposals: calibrates δ per class to a decidable-harm false-pass rate < 1 %, validates the measured-depth `AttackCost̂` mechanism of D-4 against **profitable** manipulation, and sizes POL. The proposal population's true-effect magnitude is modelled independently of δ (a proposal's welfare effect is a fact about the proposal, not the market's decision margin), so δ is a genuine calibration lever. The `3·InCapPrize ≤ AttackCost̂` step-9 certificate (SF = 3 kernel margin) and the [05](05-welfare-and-decision-engine.md) §5.6 `ManipFloor̂ ≥ 3·InCapPrize` Phase 3–4 sizing target are unchanged; the Phase-0 simulation additionally measures the *realized* attacker cost of each causal wrong-PASS flip and scores a flip that succeeds below the `3·InCapPrize` **budget** as a security failure **only when that realized cost is below the prize** — a profitable capture, the irreducible D-4 economic line that the SF = 3 margin conservatively guards. Causal decision-book displacement cost rises with δ, so δ-calibration closes profitable displacement captures (it moves the surviving near-boundary marginal exploit from ≈0.47× to ≈2.49× the prize); a flip whose realized cost instead holds **≥ the prize** — thin-market / gate-suppression griefing a deep-pocketed attacker sustains at a guaranteed loss — is a [14](14-threat-model.md) **TH-6** accepted residual (TM-18 is that row's superseded legacy label), recorded as a diagnostic rather than a gate failure. Public testnet contests with paid bounties for wrongful PASS, oracle corruption, or invariant breach — scope explicitly includes the frontend classes (tampered bundle, malicious provider, forged snapshot). ≥ 2 independent audit firms (scope A: ledger + constitution + guard; scope B: markets + decision + oracle) plus a mechanism-design review; scope-A re-audit before the final phase gate on any ledger change. Standing bug bounty from genesis.

**Reading of the < 1 % criterion (normative; amended 2026-07-20, SQ-269/SQ-273).** Two qualifiers on that threshold are load-bearing and neither may be dropped when the criterion is restated elsewhere ([09 §7.1](09-execution-upgrades-and-rollout.md) governs the phase gate).

1. **Per class, independently.** The rate is evaluated for **each** binding class — PARAM, TREASURY, CODE, META — and **every** class must hold strictly below 1 %; an aggregate or population-weighted rate is not the criterion. δ is calibrated per class, so a per-class target is what makes it a calibration lever at all, and a pooled rate would let a well-behaved class mask a mis-calibrated one. Each class therefore requires its own planted ground truth in the simulated proposal population.
2. **Decidable-harm, not raw.** The measured quantity is the **decidable-harm** false-pass rate — wrong PASSes among proposals whose true harm the decision rule was in a position to resolve — not the raw wrong-PASS rate over all proposals. The two differ materially and the distinction is the difference between a passable and an unpassable gate; a consumer field carrying this value is the decidable-harm rate regardless of any shorter name it is given.

**Realized-cost line (normative; ratified 2026-07-20, SQ-273).** The 1× *profitability* line stated above — a causal wrong-PASS flip is a security failure only when the attacker's realized cost falls **below the prize** — is the Phase-0 exit threshold. A flip whose realized cost holds at or above the prize but below `3·InCapPrize` is unprofitable griefing, recorded as a diagnostic and accepted as the [14](14-threat-model.md) TH-6 residual — whose own residual text names gate suppression alongside the deep-pocketed off-system-payoff attacker, so the reading covers the observed griefing mix and not merely the off-system case. The `3·InCapPrize ≤ AttackCost̂` step-9 certificate ([05 §5.6](05-welfare-and-decision-engine.md); D-4 states the same relation as `InCapPrize ≤ AttackCost̂/3`) is the **runtime** margin and is unchanged: it conservatively guards the 1× line the simulation validates directly. Raising the exit threshold to 3× is therefore not a tightening of the same measure but a different criterion, one the calibrated δ schedule alone cannot satisfy.

## 5. Cross-repo test artifacts (X-15)

The frontend's compatibility controls (§4.8) previously had no defined input feed. The publication contract below is normative; the exact schema and location constants of artifacts (1)–(4) are frozen in [02-integration-contract.md](02-integration-contract.md) §11 (the chain↔frontend runtime surface), while artifact (5) — the reference-model vector corpus — is schema-owned by [04-markets-and-pricing.md](04-markets-and-pricing.md) §5: 02 was finalized without a corpus row, deliberately, because the corpus is a test-time input consumed by suites, never by the running frontend.

**Per-release publication (backend duty).** For every tagged runtime release, backend CI MUST publish, as content-addressed release artifacts:

1. the reproducibly built **runtime wasm** with its hash, and the **SCALE metadata** with its metadata hash (the descriptor-generation inputs);
2. the **chain-spec artifacts** for every live environment;
3. the **Chopsticks configuration and Zombienet topology** files the backend's own §4.7 suites ran against;
4. **chainHead fixture transcripts** (deterministic JSON-RPC recordings) covering every storage item, event, and runtime-API call in the integration contract's critical surface;
5. the **reference-model JSON vector corpus** of §4.4 (LMSR V1–V6 + MPFR corpus + decision/ledger scenario vectors), as regenerated by that release's CI.

Artifacts are published to the backend repository's release feed and mirrored content-addressed (Arweave) so frontend release verification never depends on a mutable host.

**Publication ordering and the mirror duty (normative; added 2026-07-20).** "Published" and "mirrored" above are two steps, not one, and their order is forced: CI publishes but cannot mirror, because CI holds no Arweave keys ([12 §1.4](12-release-and-operations.md)). The ordering is therefore (a) backend CI publishes the artifact set to the release feed as a **prerelease** whose notes state that it is not canonical and whose manifest records the mirror duty as outstanding; (b) an operator mirrors the complete set content-addressed and attaches **mirror evidence** — a mapping from every published content digest to its Arweave transaction id; (c) the release is promoted to canonical. **"Publishable" in the gating rule below is step (a)** — the gate a runtime release must clear to ship at all. Canonical status, which is what a frontend release may depend on under §4.8 and what [12 §1.4](12-release-and-operations.md) may repoint to, requires (c). A prerelease that never obtains mirror evidence never becomes canonical, so the mutable-host independence asserted above is preserved by (c) rather than by (a).

**Gating rule (binds both WBS rows).** Backend row **E15** and frontend row **FE-R1** implement the two sides of this contract; both are **release-gating**: a runtime release is not publishable until its artifact set (1)–(5) is published and the backend's own suites ran against exactly those artifacts; a frontend release is not publishable until §4.8's descriptor-drift, fixture, Chopsticks and Zombienet suites are green **against the published artifacts of every runtime version in its supported range**. A runtime change that breaks `FutarchyApi` or critical-surface compatibility MUST NOT merge without a coordinated frontend release (old P-10, accepted by D-2), and the D-14 descriptor lead time (`DescriptorLeadTime`, *normative value: [13](13-parameters.md)*) gives the frontend train the enforced window to consume a new artifact set before enactment.

**What "covering" means for artifact (4) (normative; added 2026-07-20).** A release-time fixture is recorded against a freshly built chain, which has not exercised the protocol; requiring it to contain an instance of every frozen event would make artifact (4) unproducible. Coverage of the critical surface is therefore defined per entry as (i) the entry is **present in this release's metadata** under its frozen name, and (ii) its **rendered layout agrees** with the layout [02 §11](02-integration-contract.md) freezes — the drift check the frontend's descriptor suites actually consume. Additionally, once per fixture, (iii) the `System.Events` blob of a **real executed block** MUST be retrievable and non-empty over **both** the `chainHead` and the legacy storage transports, so a fixture cannot certify a surface that only one transport can reach. Requirement (iii) is a single per-fixture transport-and-liveness proof and explicitly **not** per-variant evidence: a release fixture does not assert that any particular event variant was ever emitted. Demonstrating actual emission belongs to the scenario-driven §4.7 suites, whose execution is evidenced by the environment-run contract below.

**Environment run evidence (normative; added 2026-07-20).** "The backend's own suites ran against exactly those artifacts" is discharged by a machine-checkable artifact, never by a green commit-level CI run: commit CI proves the *source* was tested, not that the *packaged bytes* were. The release pipeline MUST re-execute the suites against the built artifacts and emit run evidence carrying, as frozen fields: the schema identifier; the suite kind; the **release runtime-wasm hash**; the **release commit**; the **tier** the run covers (below); the **named suites that ran, each with its result**; the **suites excluded, each with its reason** (below); and a **per-file digest inventory** of the environment definitions the run used. Assembly MUST validate that evidence against the packaged artifact set **byte-for-byte** — every inventoried file present and hash-equal, with no unpackaged extras — and MUST reject an **empty** inventory, which would otherwise let a placeholder environment directory satisfy the gate vacuously. It MUST also refuse evidence whose wasm hash or commit does not bind to the release under assembly. Producers MAY emit additional fields; consumers MUST NOT depend on them. A release whose evidence is absent or invalid is not publishable — including when the evidence is withheld because a suite's prerequisites are unwired, which is the intended fail-closed behavior and not a reason to relax the gate.

**Suite tiering (normative; added 2026-07-20).** §4.7's list includes drills whose real-time cost — multi-hour finality stalls, multi-day relay-block leads, multi-epoch soaks — makes per-tag execution physically impossible. The gating rule is therefore satisfied against a declared **release tier**: the subset of §4.7's suites that MUST run against the packaged artifacts at every tag, with the remainder assigned to a **phase-gate tier** proven against the [09](09-execution-upgrades-and-rollout.md) §7.1 rollout gates instead. This is a deliberate narrowing of the gating sentence above — §4.7 names the dead-man drill among the backend's own suites, and its real-time cost puts it in the phase-gate tier, so a tag's evidence does not cover it — and it is admissible only under all four of: (1) tier membership is declared in a checked-in manifest and **pinned by the environment validator**, so no suite changes tier silently; (2) a release-tier suite that does not run and pass **blocks evidence emission** — it never degrades to a pass, whatever the reason for the skip; (3) every excluded suite is **recorded inside the evidence** with its reason — a frozen field, validated at assembly, so the subset a release actually ran is auditable from the artifact alone rather than resting on a producer's goodwill; and (4) evidence is admissible for this gate only when stamped with the release tier. Tier membership is a claim about execution cost and never about importance: a suite sits in the phase-gate tier only because it cannot run per tag, and [09](09-execution-upgrades-and-rollout.md) §7.1 owns proving it.
