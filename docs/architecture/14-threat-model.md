# 14 — Combined System Threat Model

**Status: normative component specification. Supersedes the corresponding sections of BACKEND_PLAN.md/FRONTEND_PLAN.md** (BE §22, FE §20).

**Boundary.** This document owns the combined threat model for the whole system — chain, protocol economics, governance, release pipeline, and frontend — as one adversary surface. It defines the adversary model, enumerates every threat row, and records which threats are closed, mitigated, or accepted residuals. It does **not** own any mitigation mechanism: every control cited here is specified normatively in the linked component document. All numeric constants quoted in rows are *(normative values: [13-parameters.md](13-parameters.md) and [02-integration-contract.md](02-integration-contract.md))*.

RFC 2119 language applies. Any new pallet, screen, or operational role added to the system MUST add or amend a row here in the same change set; a threat row MUST NOT be deleted — closed threats are retained with status **CLOSED** and the original attack described, so the record of why a control exists is never lost.

---

## 1. Adversary model

Uniform assumptions (union of BE §22 and FE §20, extended per [D-13]):

- The adversary is rational and well-funded; it can propose, trade, crank, report, dispute, vote, run collators (within the phase's collation rules), run gateways, indexers, RPC endpoints, bootnodes and snapshot publishers, publish frontend forks, and attempt XSS/supply-chain compromise.
- The adversary **cannot** break relay-chain consensus (beyond FE assumption A-5's scope), SR25519/ed25519, BLAKE2/SHA-256, or SCALE encoding, and cannot forge GRANDPA justifications.
- **Insiders are in scope.** The adversary model explicitly includes: the **4-of-6 founding multisig** while `pallet-sudo` exists (Phases 0–3 — this inclusion is mandated by [D-13]/X-9); release-key holders; ArNS ANT controllers; collators (including a colluding majority of the launch set); oracle reporters; watchtowers; attestors; guardians. No single insider class is assumed honest — each has a row below.
- Frontend boundaries (recap): device (browser sandbox, extension IPC), distribution (content addressing), chain access (light-client verification), acceleration (package firewall + provenance typing), human (labelling + education).

**Phase legend** (rollout phases per [09-execution-upgrades-and-rollout.md](09-execution-upgrades-and-rollout.md)): 0–1 dev/testnet · 2 shadow operation (no real USDC) · 3 guarded mainnet (real USDC under sudo, exposure-capped) · 4 sudo removed, permissionless collation · 5 fully armed treasury. "all" = every phase with the relevant surface live.

## 4. Accepted residuals (system-level summary)

These are the residuals this model *accepts and states honestly* rather than claims to eliminate. They MUST be carried into user-facing honesty surfaces where applicable ([11](11-frontend-workflows.md)) and re-reviewed at every phase gate ([09](09-execution-upgrades-and-rollout.md)):

1. **Phase-3 trust phase** (TH-29): sudo can upgrade the runtime; exposure caps bound the loss, the banner makes it visible. Closes at Phase 4.
2. **Watchtower + collator collusion** (TH-24/TH-25): the only remaining path to a *wrong* (not merely delayed) censored settlement.
3. **Plutocracy in the values lane** (TH-19) and the A-3 trust root (TH-22): supermajorities and conviction, not prevention.
4. **Off-system-payoff attacker** (TH-6): in-system prizes cannot price external payoffs; caps and fork doctrine bound it.
5. **Ordinary fill MEV** (TH-3).
6. **Hostile service worker** (TH-45): in-band detection is unsound; out-of-band monitoring + manual recovery only.
7. **Provider chart / deep-history forgery** (TH-49/TH-50): provenance labels and the transaction firewall, not detection; sampling verifies almost nothing at depth.
8. **Signer disjointness is organizational** (TH-41), and ArNS controller policy is social, not protocol (TH-39).
9. **Perfect-clone phishing relaying the real chain** (TH-46).
10. **Reserve freeze is honest, not reversible** (TH-14); depeg value risk sits with holders (TH-13).
11. **Goodhart arms race** (TH-5) and in-envelope parameter drift (TH-20).
12. **A-2 (arbitrage elasticity) is empirical** (TH-1) and MUST be re-validated at each phase gate via the adversarial economic simulation ([15](15-invariants-and-testing.md)).
