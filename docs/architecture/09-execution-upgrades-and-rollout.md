## 1. Execution guard (`pallet-execution-guard`)

The guard is the single belief-side execution path. `pallet-scheduler` is deliberately absent from it: scheduler agendas capture origin at scheduling and would not repeat the checks of §1.2; the guard is the revalidation point. Belief-side execution never touches the scheduler; the scheduler remains solely `pallet-referenda`'s enactment engine, whose dispatches re-enter `BaseCallFilter` + origin checks at dispatch time (carried from BE §6.4; the phantom "§18.6" references in both source documents are void — the attestation regime is §2.4 of this document).

### 1.1 Queue semantics and queue-time preconditions

`enqueue` is callable only by `pallet-epoch`'s decision path (I-9). Storage:

```
QueuedExecution {
  pid, payload_hash, payload_len, class, maturity, grace_end,
  version_constraint, meters_declared, ratify_ref: Option<ReferendumIndex>,
  ratification_passed: bool,                   // §1.2(4); projected into 02 §4's RatificationStatus
  declared_domains: Vec<CallDomain>,           // §1.2(11) domain-subset check
  failed_at: Option<BlockNumber>,              // 05 §2.1 T18; Some(f) marks FailedExecuted
  attestation_id: Option<AttestationId>,       // CODE/META only
  // (retired 2026-07-20, SQ-127/SQ-144: the PB-MIGRATION anchor is captured at
  //  code *application*, not at execute(), and single-homed in its own guard
  //  storage item — this row is deleted by §1.2(13) before a migration can fail.
  //  See §3.2(2). NOTE: the shipped struct retains an inert
  //  `pre_upgrade_checkpoint` for the frozen v6 SCALE shape and always writes
  //  `None`; physical removal remains a coordinated storage change owed under
  //  02 §13(3) and tracked in PLAN.md · Spec questions (SQ-94).)
  cancelled: bool
}
```

Queue bound = `MaxLiveProposals = 32` *(normative value: [13](./13-parameters.md))*.

Queue-time preconditions (enforced by the decision path before `enqueue` succeeds):

1. `decide()` returned Pass with all gate/welfare checks per [05](./05-welfare-and-decision-engine.md) — including the decide-time `SecuritySizing` outflow cap (D-4).
2. **Preimage pinned**: the payload preimage was pinned at qualification via `request_preimage` (§7.3) and is still noted. A missing preimage at queue time cancels per T4 with the 10% bond slash owned by [06](./06-governance-and-guardians.md)/[08](./08-treasury-and-economics.md) (B-13).
3. **CODE/META only — kernel attestation complete (D-18 upgrade of BE §18.4)**: a valid `AttestationRecord` for the committed artifact hash exists with **≥ 2-of-3 attestor signatures** from the bonded attestor registry and its **72h challenge window** *(normative value: [13](./13-parameters.md))* has elapsed with no challenge resolved against the record (§2.4). Attestation is achievable well before queue time because the artifact hash is committed at proposal submission and the trading window is 13 days.
4. **CODE/META only** — the `ratify` referendum reference is recorded (`ratify_ref`). Per D-5 there is **one** ratification deadline and it is checked at `execute()` dispatch time, not here; the referendum runs during the timelock. Recording the reference at queue time merely binds the proposal to a specific referendum.

### 1.2 `execute(pid)` — permissionless, atomic; the complete dispatch-time check list

Checks in order, **all at dispatch time**. This list is normative and canonical: the FE `execute` precondition row in [11](./11-frontend-workflows.md) MUST mirror it item-for-item (X-11i — the source FE row omitted items 6–10 below).

1. **Queue state**: queued, not cancelled, `maturity ≤ now ≤ grace_end`.
2. **Preimage**: fetched; hash and length match the trading-time commitment (pinned since qualification, §7.3).
3. **Version constraint**: `RuntimeVersionConstraint` matches live `spec_name`/`spec_version` — mismatch (an intervening upgrade) transitions to `Rejected(StaleQueue)`, requiring resubmission. **Layout (normative here; the type appears inside the frozen [02 §4](./02-integration-contract.md) `QueuedExecutionView` and the doc-05 `Proposal`):** `RuntimeVersionConstraint { spec_name: BoundedVec<u8, 32> /* UTF-8, = RuntimeVersion.spec_name */, spec_version: u32 }`.
4. **Ratification (D-5)**: for CODE/META (and rule-altering META per the [06](./06-governance-and-guardians.md) capability table), a Passed `RatificationRecord` bound to `(pid, payload_hash)` is REQUIRED. **If it is absent or its referendum has not passed, `execute(pid)` errors `NotRatified` and the proposal remains `Queued`** — keepers MAY retry through `grace_end`, and the referendum may still pass in the meantime. Only the post-grace T16 crank turns that into `Rejected(NotRatified)` ([05](./05-welfare-and-decision-engine.md) §2.1; SQ-163 resolution, 2026-07-21 — the superseded wording said the dispatch-time check itself rejects, which would have made a single early keeper attempt terminal and thrown away the timelock the referendum is meant to run inside). This is the single ratification deadline; there is no queue-time ratification check.
5. **Attestation presence**: for CODE/META, the `AttestationRecord` referenced by `attestation_id` still exists, is unrevoked and unchallenged → else `Rejected(AttestationMissing)`. (Content validity was established at queue time, §1.1(3); this check catches post-queue revocation by a challenge resolved against the attestation.) **This check is over the record, never the roster (normative; SQ-97 resolution, 2026-07-20):** it MUST NOT additionally require that the attestation's signer still sits in the attestor registry, and it MUST NOT re-evaluate the ≥ 3-active-member floor or the distinct-active-signer count — those are *admission* conditions that bind at **queue time** ([06](./06-governance-and-guardians.md) §7, *What binds where*). A signer's departure is not evidence about the artifact, whose hash is pinned at qualification (§7.3); a challenge is, and a challenge is what this check reads. **This relaxation is NOT in force and MUST NOT be implemented yet.** It is conditional on [06](./06-governance-and-guardians.md) §7's for-cause revocation rule, which in turn requires a cause-carrying removal surface the frozen [02 §7.5](./02-integration-contract.md) does not provide. Shipping the relaxation without it would let an upgrade execute on attestations from a signer the values layer removed for cause but could not adjudicate — strictly weaker than the live-registry check it replaces. **Until that surface exists, the shipped live-registry check at dispatch stands**, together with the liveness cost 06 §7 records. See PLAN.md · *Spec questions* for the two ways out; the choice is an integration-contract decision, not a doc-09 one.
6. **Capability rules**: every call domain in the batch is admissible for the class origin per the constitution capability table.
7. **Rate meters**: treasury-outflow, issuance and `code.spacing` meters admit the batch (I-7, I-17). Meter contention ⇒ execution stays queued and retries within grace.
8. **Resource locks**: all declared resource-domain locks are still held by `pid`.
9. **No guardian hold**: no active `delay_once` on `pid`, no guardian suspension touching the queue.
10. **Gate flags / freezes**: no active hard-gate daily breach flag, no dead-man freeze, no active PB-LEDGER-FREEZE and no PB-MIGRATION halt-at-fault (§3.2). The enumerated coretime-renewal call is the sole exemption and does not pass through the queue at all (§5).
11. **Payload bounds**: decode batch (≤ 16 calls, ≤ 64 KiB, declared weight ≤ 25% of block limit, dispatch class Normal); re-derive call domains and assert ⊆ declared domains (I-11); recursively apply the **closed** SafetyFilter (§1.4) to nested wrappers. **The decode itself is recursion-bounded (normative; SQ-225 resolution, 2026-07-20):** the guard MUST decode the preimage-sourced batch with a SCALE depth limit of `MAX_PAYLOAD_DECODE_DEPTH` *(normative value: [13](./13-parameters.md) §2)* and MUST require full consumption of the committed bytes. An over-deep or trailing-byte-bearing preimage resolves to `Rejected(BadPreimage)` (G-1 status quo) and MUST NEVER reach a wasm stack-height trap. `MAX_NESTED` ([06](./06-governance-and-guardians.md) §3.3) is a **post-decode** filter bound and therefore cannot bound the decode — the two limits are independent, and this one is the audit-scope-A one.
12. **Dispatch**: each call with the class origin (`FutarchyParam`/`FutarchyTreasury`/`FutarchyCode`/`FutarchyMeta`); the batch is atomic — any item error rolls back the batch's state and transitions per T18.
13. **Record**: write `ExecutionRecord`, consume meters, release locks, unpin the preimage, emit `Executed`.

### 1.3 Origin discipline

Every class origin is produced by exactly one pallet through exactly one code path; none is obtainable from a signed extrinsic, XCM origin conversion, or wrapper call. There is no path to unrestricted Root: `EnsureRoot` succeeds only for the internal dispatch the guard performs for the two allowlisted `frame_system` upgrade calls, constructed inside the runtime (I-10).

### 1.4 SafetyFilter closure — execution-side enforcement

The SafetyFilter recursion rule is owned by [06](./06-governance-and-guardians.md). Normative consequence enforced here: the wrapper/carrier set is **closed and single-homed in [06](./06-governance-and-guardians.md) §3.3**, and it covers typed wrappers *and* reviewed metadata-opaque semantic carriers (SQ-153). It MUST include `proxy_announced` and `as_multi_threshold_1` alongside `batch`/`batch_all`/`force_batch`/`proxy`/`as_multi` (the review's completeness finding: had they fallen to allow-without-recursion, the filter was bypassable), and the denied carriers `utility.if_else`, `utility.dispatch_as_fallible`, `sudo.sudo_as` and `pallet_xcm.execute`. The phrase "the only two unhandled call-wrappers" described the state at that review and is no longer true of the runtime. Guard step 11 applies the closed filter to every nested level (`MAX_NESTED = 4` levels deep, ≤ 16 calls total counted as **nodes** — counting rule owned by [06](./06-governance-and-guardians.md) §3.3); [15](./15-invariants-and-testing.md) carries the negative-test obligation (I-10/I-11 fuzz over *all* wrapper types, including the two added ones, and over `sudo.sudo(call)` during Phases 0–3 per §6.1).

### 1.5 Execution records and history serving

`ExecutionRecords` is a ring buffer (≤ 256 entries). The source's "pruned to indexer" language is **replaced** (D-2/D-6): canonical execution history is **event-derived and chain-served within the committed window** — `Executed`/`Rejected(*)`/`UpgradeAuthorized`/`UpgradeApplied` events plus the ring buffer are servable via the [02](./02-integration-contract.md) three-layer history model (chain-served recent window; 30-day committed operator window). The indexer is an optional dashboard convenience, **never load-bearing** for the canonical frontend.

## 2. Runtime-upgrade path (CODE or META class)

### 2.1 Two-phase authorize/apply flow

1. Proposer commits the artifact set at submission: candidate Wasm **hash**, target `RuntimeVersion` (spec_version N+1), **metadata hash** (via `frame-metadata-hash-extension` tooling — verified crate), migration plan with benchmarked migration weight, audit-report hash, and kernel **attestation reference** (§2.4). The payload is exactly `system.authorize_upgrade(code_hash)` (verified dispatchable) plus the commitment record write, plus the `ReleaseChannel` update (§2.3).
2. Markets trade the committed artifact; gate markets mandatory; the `ratify` referendum is submitted any time after the artifact-hash commitment and runs during the timelock (D-5).
3. After the timelock, `execute(pid)` runs the full §1.2 list and dispatches `authorize_upgrade(code_hash)` with internally constructed Root restricted to this single call and hash (I-10, I-19). The guard records `PendingUpgrade { hash, authorized_at: now, applicable_at: now + DescriptorLeadTime }`, updates `ReleaseChannel` by setting `pending_authorized_at` and `URGENT_UPGRADE` **without changing its current-runtime `spec_version`** (§2.3), and emits `UpgradeAuthorized { code_hash, authorized_at }` — the two-field shape frozen by [02 §6](./02-integration-contract.md), which this list previously contradicted by naming a third field (`applicable_at`); it is derivable as `authorized_at + DescriptorLeadTime` and is not carried (doc-truing, 2026-07-20). The PB-MIGRATION anchor is **not** written here — it is captured at code application (§3.2(2)). `authorize_upgrade_without_checks` and `set_code*` are filtered for every origin, always, from genesis (§6.1).
4. **Application is lead-time-gated (D-14, §2.2).** From `applicable_at`, **anyone** submits the matching artifact via `system.apply_authorized_upgrade(code)` (verified); `cumulus-pallet-parachain-system` performs the parachain code-upgrade signaling to the relay **[VERIFY current go-ahead mechanics on stable2606]**.
5. Migrations run under `pallet-migrations` (verified crate): multi-block, cursor-bounded, benchmarked; `try-runtime` + `try-state` gate the release in CI.
6. Post-upgrade: the guard records the observed `spec_version`, writes that newly installed version to `ReleaseChannel`, and unconditionally clears the channel pending indication and `PendingUpgrade`; failed post-upgrade checks on testnet block the release; on mainnet, migration failure follows `pallet-migrations` halt-and-alarm semantics — a `Stuck` cursor bridged to the runtime's `MigrationHalt` execution-queue freeze via the halt-source sync (verified on stable2606, SQ-274) — with the now fully specified playbook **PB-MIGRATION** (§3.2) as recovery.
7. **Relay abort (normative; SQ-131 resolution, amended by contract v6).** If the relay signals `UpgradeGoAhead::Abort`, `cumulus-pallet-parachain-system` deletes `PendingValidationCode` **without any runtime callback**, after `frame-system` has already consumed the authorization (verified on the `stable2606` pin, `cumulus-pallet-parachain-system` 0.29.0). Left unhandled, the guard's `PendingUpgrade` would wedge every future upgrade. The runtime MUST therefore detect the abort deterministically in the validation-data hook — scheduling latch plus an installed-code-hash mismatch — and then: clear `PendingUpgrade` and the §3.2(2) capture latch — no pre-migration anchor exists on this path, since capture happens at application and an abort precedes it; **unconditionally** clear `ReleaseChannel.pending_authorized_at` and `URGENT_UPGRADE` while leaving `ReleaseChannel.spec_version` untouched, because the old runtime remains installed (§2.3); emit append-only `UpgradeAborted { code_hash }`; and raise a guardian-visible incident bit that does **not** by itself freeze the execution queue (§3.2). The artifact is **re-proposable through the normal process**; there is no automatic re-authorization — the relay declined it, and a status-quo default (G-1) does not retry a rejected upgrade on the chain's own initiative. A failed cleanup on this path raises the PB-MIGRATION halt (§3.2).

Edge handling (carried): queued-under-N-executed-after-upgrade ⇒ rejected by check §1.2(3); storage-version mismatch ⇒ migration framework refuses ⇒ PB-MIGRATION; excessive migration weight ⇒ multi-block spreading within declared bound, else abort ⇒ PB-MIGRATION; metadata incompatibility ⇒ wallet-detectable via metadata hash; emergency rollback: **none on-chain** — a rollback is a forward upgrade through the same process (or the expedited lane, §3.1); collator binary compatibility handled by the ops release train ([12](./12-release-and-operations.md)).

### 2.2 `DescriptorLeadTime` (D-14, resolves X-7)

`system.apply_authorized_upgrade` for a CODE/META artifact MUST NOT be dispatchable before

```
now ≥ authorized_at + DescriptorLeadTime        // 43,200 blocks = 72 h (normative value: 13)
```

where `authorized_at` is the block of the `UpgradeAuthorized` event (guard step §2.1(3)). Enforcement is deterministic and origin-independent: the SafetyFilter consults `PendingUpgrade` and denies `apply_authorized_upgrade` while `now < applicable_at`; an early submission fails statelessly. This closes the X-7 defect — application could previously follow authorization by one block, opening a ≥ 72h global `ReadOnlyIncompatible` signing outage that outlasted 48h obligations (oracle challenge windows now 72h with watchtower quorum, [07](./07-oracle-and-disputes.md), interlock preserved: the challenge window never fits inside an un-warned descriptor gap).

The frontend obligation is the mirror half and is **release-gating, not a convention**: v(N+1) descriptors MUST be generated from the queue-time artifact commitment and live on the release channel **before `applicable_at`** ([12](./12-release-and-operations.md)); the expedited descriptor-only release (2 attestations, no 72h soak, 3-of-5 repoint) exists for exactly this window. Because the artifact hash and metadata hash are committed at *submission* (≥ timelock + trading window before `applicable_at`), the 72h lead time is a floor on the last mile, not the whole descriptor budget.

### 2.3 `ReleaseChannel` — fixed-layout raw storage key (D-14; replaces the FE §24.5 `system.remark` pointer)

`pallet-constitution` exposes a `StorageValue` at the **well-known raw key**

```
twox_128("Constitution") ++ twox_128("ReleaseChannel")
```

whose fixed-width byte layout is **frozen forever** — it MUST never change across any runtime upgrade; any future extension appends fields beyond offset 168 with a schema bump, never a change to existing offsets or a rename to a new key. This makes the value readable by stranded apps (pinned releases in `ReadOnlyIncompatible`) that cannot decode current metadata: static key, fixed offsets, no metadata required. **The byte layout is owned and frozen by [02 §12](./02-integration-contract.md)** (168 bytes; quoted for readability — *normative layout: 02 §12*):

```
ReleaseChannelRecord {                 // normative layout: 02 §12 (offsets there)
  schema:                u8,          // constant 1
  version:               [u8; 32],    // UTF-8 semver, zero-padded — current canonical release
  manifest_txid:         [u8; 43],    // Arweave base64url TXID, zero-padded
  release_json_hash:     [u8; 32],    // SHA-256
  updated_at:            u32,         // block of last update
  spec_version:          u32,         // current runtime spec_version
  pending_authorized_at: u32,         // block of a pending UpgradeAuthorized; 0 if none
  min_supported_version: [u8; 32],    // UTF-8 semver, zero-padded (12 §3.2)
  keyring_generation:    u32,         // 12 §2.1
  revoked_key_bits:      u64,         // revocation bitmask, 12 §2.3
  flags:                 u32,         // bit 0 SECURITY, bit 1 EXPEDITED, bit 2 URGENT_UPGRADE
}
```

Writers and field ownership, exhaustively and normatively: (a) the execution guard **exclusively owns** offsets 112–119 (`spec_version`, `pending_authorized_at`) and flag bit 2 (`URGENT_UPGRADE`). At `UpgradeAuthorized` it sets `pending_authorized_at` and bit 2 but leaves `spec_version` untouched; at applied-upgrade detection it sets `spec_version` to the newly installed version and clears the pending fields; at **relay-abort cleanup** (§2.1(7)) it clears the pending fields and leaves `spec_version` untouched. Both terminal clears are unconditional on the previous channel contents. (b) `ConstitutionalValues` via `constitution.set_release_channel` owns offsets 1–107 and 120–163 plus flag bits 0–1 for manifest-pointer, `min_supported_version` and keyring-generation/revocation updates, including expedited descriptor-only releases ([12](./12-release-and-operations.md)); it **MUST preserve** writer (a)'s bytes and bit, and the call merges the caller's record with those stored guard-owned fields. Offset 0 remains the fixed schema byte. No other origin can write the record. The FE in-app upgrade warning for pinned-release users binds to this key ([12](./12-release-and-operations.md)).

**`updated_at` (offset 108) is shared write metadata written by every writer (normative; SQ-133 resolution, contract v6 ownership amendment).** The field records the block of the **last write to the record**, not the last manifest repoint: writer (a) stamps it at authorization, applied-upgrade detection and relay-abort cleanup; writer (b) stamps it on each of its writes. These ownership ranges are exclusive except for this shared field: the writer (b) merge MUST ignore any caller attempt to alter offsets 112–119 or flag bit 2, and writer (a) never touches the manifest, minimum-version or keyring/revocation fields. Manifest freshness is carried by `manifest_txid`/`release_json_hash`; monitoring that needs repoint lag MUST derive it from the manifest pointer, never from offset 108. The invariant `PendingUpgrade.is_some() ⇔ pending_authorized_at != 0 ⇔ URGENT_UPGRADE` is I-30 ([15](15-invariants-and-testing.md)) and is enforced by execution-guard `try-state`.

### 2.4 Kernel attestation — bonded attestor registry (upgrade of BE §18.4; no longer presence-only)

Deterministic Wasm builds (`srtool`-class **[VERIFY current recommended tooling]**) are carried. The presence-only attestation of the source ("any 32-byte hash passes") is replaced:

- **Attestor registry**: ≥ 3 values-elected attestors, each posting a bond; election, recall, bond size and slash routing are owned by [06](./06-governance-and-guardians.md) (registry governance) and [13](./13-parameters.md) (values).
- **AttestationRecord** per artifact: `{ artifact_hash, source_commit, build_env_digest, signatures: BoundedVec<(AttestorId, Signature), 8>, submitted_at, challenge_end }`. Valid iff ≥ **2-of-3** registered attestors sign that they independently reproduced `artifact_hash` from `source_commit`.
- **Challenge window**: 72h *(normative value: [13](./13-parameters.md))* from record submission. A bonded challenge alleging non-reproducibility routes through the [07](./07-oracle-and-disputes.md) dispute game (deterministic recomputation is the natural terminal step: rebuild and compare hashes); a challenge **resolved against the attestation** slashes the signing attestors, revokes the record, and any queued proposal referencing it fails §1.2(5) at execute. **Polarity (normative; SQ-97 resolution, 2026-07-20):** the frozen [02 §7.5](./02-integration-contract.md) enum `ChallengeStatus` names the verdict from the *attestation's* side — `Upheld` means the **attestation** was upheld and the challenger forfeits; `Rejected` means the attestation was struck and its signers forfeit. The doc set was trued to this polarity in the same pass (05 §1.3/§2.1/§5.5, 15 I-19, and this section); should any older prose survive, read "an upheld challenge" there as `ChallengeStatus::Rejected`. Only `Rejected` and `Open` withhold an attestation from quorum.
- **Gating**: record validity + elapsed window is a **queue-time precondition** (§1.1(3)); record continued existence is a dispatch-time check (§1.2(5)). Queue time evaluates the live registry (member floor, distinct active signers, closed windows). **Dispatch time evaluates the live registry as well, and MUST continue to** (normative; corrected 2026-07-21, SQ-312): §1.2(5)'s narrowing of the dispatch check to the committed records' revocation/challenge status alone is the *conditional* SQ-97 relaxation, and it is **NOT in force** — it is blocked on the for-cause revocation rule of [06](./06-governance-and-guardians.md) §7, *What binds where*, which needs a cause-carrying removal surface the frozen [02 §7.5](./02-integration-contract.md) does not provide. This bullet previously stated the relaxed split flatly, contradicting both §1.2(5) and 06 §7 and misdescribing the shipped runtime. The two checks therefore read the **same** state today; they will diverge only when that surface lands.

Every release still publishes source commit, build-environment digest, Wasm hash, metadata hash. Verifying attestation *content* beyond hash reproduction remains social/off-chain (A-4) and is stated as such — the registry converts "presence-only" into "bonded, plural, challengeable", not into a mathematical proof.

### 2.5 Worked example (BE §30.4, updated)

META proposal commits (wasm_hash H, spec 102, metadata hash M, migration weight 40% of one block spread over 3 blocks, audit hash, attestation reference A). Attestors 1 and 3 sign A during the first trading week; the 72h challenge window closes unchallenged; queue-time preconditions hold. The values `ratify` referendum passes day 12 (during the timelock). Markets: uplift 0.095 ≥ 0.090 (V-12 `δ_META`); gates clean. Queued, timelock 14 d. `execute` passes checks 1–11 (ratified ✔, attestation present ✔, meters ✔) and dispatches `authorize_upgrade(H)` via restricted internal Root; `UpgradeAuthorized{H, authorized_at}`; `ReleaseChannel` updated. No anchor is recorded here — the PB-MIGRATION anchor is captured at code application (§3.2(2)). The FE ships v(N+1) descriptors (generated from the queue-time commitment) inside the 72h; the release-gating check in [12](./12-release-and-operations.md) confirms them live. At `applicable_at + 10`, a keeper submits `apply_authorized_upgrade(code)`; parachain-system schedules the swap; migrations run over 3 blocks; guard records spec 102; `try-state` clean. A stale PARAM item queued under spec 101 later hits §1.2(3) → `Rejected(StaleQueue)` and is resubmitted.

## 3. Emergency lanes

### 3.1 Expedited CODE lane (D-9, resolves B-17 lane side)

A repair upgrade MUST NOT need a new privileged origin, and it does not get one. The expedited lane is the normal CODE process with compressed windows, admissible **only while a machine-checked emergency condition is active**:

- **PB-LEDGER-FREEZE** is active (guardian 5-of-7 playbook keyed to the I-4 drift flag; contents, expiry ≤ 14 days + one renewal, and retro-ratification owned by [06](./06-governance-and-guardians.md)); **or**
- a **PB-MIGRATION halt-at-fault** is active (§3.2) — without this arm the designated migration fallback could not terminate, since its rollback path is a forward upgrade and the normal lane is O(25 days).

Lane mechanics:

| Aspect | Expedited rule | Normal-lane rule |
|---|---|---|
| Admissibility | one of the two freeze conditions active, machine-checked at submission *and* at decide | always |
| Gate markets | mandatory, **72h** decision window | full trading window |
| Values ratification | fast-track `ratify` referendum, **3-day** decision period ([06](./06-governance-and-guardians.md) track table); checked at execute per D-5 | 7-day ratify |
| Attestation | full §2.4 regime (2-of-3 + 72h window, runnable concurrently with the gate market) | same |
| Execution | the **normal execution guard**, full §1.2 list; check §1.2(10) treats the *triggering* freeze as satisfied for the expedited proposal itself (the freeze exists to contain the fault the proposal repairs) | full list |
| `DescriptorLeadTime` | **applies in full** — the freeze contains the exploit, so the 72h descriptor floor is affordable; the expedited descriptor-only FE release ([12](./12-release-and-operations.md)) covers the last mile | applies |
| `code.spacing` meter | **waived for the expedited proposal** (a 14–30 d spacing floor inside a ≤ 14-day freeze would nullify the lane); the meter still *records* the upgrade, and the next normal-lane CODE measures spacing from it | enforced |
| Privileged origin | none — dispatches as `FutarchyCode` | none |

Worst-case repair latency under an active freeze: 72h gate + 3d ratify (concurrent where windows overlap) + timelock floor 24h + 72h lead time ≈ **9–10 days**, inside the 14-day freeze envelope — versus the ~25–46-day normal path B-17 measured.

### 3.2 PB-MIGRATION — contents (resolves the empty-designated-fallback finding)

PB-MIGRATION is a registered guardian playbook (preimage-committed, values-ratified in advance, scoped, expiring). The review flagged it as an empty name; its contents are now normative:

**Trigger (machine-checked, any of):** `pallet-migrations` reports a failed migration step; migration cursor **stalled** > 900 blocks (predicate defined below); **applied-code mismatch** (the code or `spec_version` observed after application is not the authorized artifact, including a failing applied-code validation); **failed abort cleanup** (a relay `UpgradeGoAhead::Abort` whose §2.1(7) cleanup did not complete). Activation requires guardian 5-of-7 while a trigger is active.

**Trigger-list amendments (normative; SQ-132 resolution, 2026-07-20).** Three corrections to the closed list above. **(a)** The former third arm — "post-upgrade `try-state` assertion failure signal" — is **retired as an on-chain trigger**: `try-state` does not run in production block execution on any Substrate runtime (every `frame-executive` call site is `#[cfg(feature = "try-runtime")]`), so the arm was unimplementable as written. Post-upgrade `try-state` remains a release gate in CI (§2.1(5)) and an operator signal via the [12](./12-release-and-operations.md) §6.3 monitoring surface; it never raises the on-chain halt. **(b)** The two arms added above (applied-code mismatch, failed abort cleanup) were already the conservative shipped behaviour and are now enumerated, so the list is closed against the runtime rather than against an unreachable ideal. A **clean** relay abort raises only the guardian-visible incident bit and does **not** freeze the execution queue (§2.1(7)); only a *failed* cleanup halts. **(c)** In item (3) below, "`try-state` must pass before the freeze lifts" binds the **guardian playbook freeze** as an operator precondition, not the on-chain `MigrationHalt` flag — which clears mechanically on migration completion or on successful recovery-image application, since (a) leaves no on-chain try-state signal to gate it on.

**(d) "Stalled" is a time budget, never a cursor-motion test (normative; SQ-132(d) resolution, 2026-07-20).** A migration is **stalled** iff an active cursor's own start block is more than `MIGRATION_STALL_BLOCKS` in the past — `now − cursor.started_at > 900` *(normative value: [13](./13-parameters.md) §2)*. It is **not** defined by the cursor bytes failing to change. The pinned SDK imposes no cursor-advance obligation: `SteppedMigration` constrains only the terminal case (a `None` cursor ends the migration) and `pallet-migrations` guarantees progress purely through `max_steps()`, measured in blocks since `started_at`. `Cursor = ()` is a first-class idiom there, so a conforming migration may drain a map for hours while returning byte-identical cursors; a byte-equality predicate false-raises on exactly that migration. The harm is **not** that it opens the expedited CODE lane — a live cursor forces `ExtrinsicInclusionMode::OnlyInherents`, so nothing can be submitted to that lane while the false halt stands (SQ-309) — but that it pages the guardian council against a healthy chain, freezes the execution queue, and invites a remediation upgrade against a migration that was progressing correctly. Two consequences bind. (i) The predicate MUST read the SDK's own `started_at` rather than a runtime-maintained progress marker, so it is a pure function of state the migrator already keeps. (ii) Every registered multi-block migration MUST declare `max_steps() = Some(n)` with `n` **strictly less than** `MIGRATION_STALL_BLOCKS`, enforced by an integrity test at build time. Strictness is load-bearing: at `n = MIGRATION_STALL_BLOCKS` the two fire in the same block and this backstop wins, because the halt check runs in the parachain inherent while the SDK compares `took > max_steps` inside the later migrator step — so only `n < MIGRATION_STALL_BLOCKS` lets the SDK's own budget fire first and raise the halt through arm 1 (*failed migration step*), leaving this arm a backstop that can only fire if that enforcement is bypassed. (The SDK's `max_steps` is documented as a step count but is compared against blocks elapsed since `started_at` in `pallet-migrations 19.0.0`; the obligation here is the block-denominated reading, and a train bump that changes it must revisit this clause.) A migration that legitimately needs longer than the budget must be split — that is what `max_steps` is for, and the constraint belongs at integrity-test time, not at block 901 on a live chain. **(iii) The aggregate run budget MUST be bounded too (normative rider; SQ-132 resolution, 2026-07-21).** Splitting a run into segments does not escape the intent of (ii): the SDK's `started_at` resets at each segment, so a multi-segment run could hold the chain `OnlyInherents` for the *sum* of the per-segment budgets while no single segment ever trips the 900-block stall. The sum of `max_steps` over **all** registered segments MUST therefore itself be strictly less than `MIGRATION_STALL_BLOCKS`, checked by the same build-time integrity test — so the whole multi-segment run cannot hold the chain inherent-only for longer than one segment's stall budget, and the aggregate is a derived quantity, inventing no new [13](./13-parameters.md) value. **Until the inherent-only recovery lane (§3.2(3), the pre-authorized recovery image applied by deterministic inherent machinery) ships, registering *any* multi-block migration is prohibited outright, enforced by that same integrity test asserting `Migrations = ()` in the production configuration.** A genuinely stuck cursor forces `OnlyInherents` and forecloses §3.2(3)'s own designated recovery path (SQ-309); with no on-chain escape from that mode yet built, the only safe number of registered migrations is zero, and the prohibition is a build-time lock, not a convention.

**1. Halt-at-fault semantics.** On trigger, the chain fail-stops the *affected surface*, not block production. For a genuine cursor, `pallet-migrations` (the runtime's `MultiBlockMigrator`) keeps **all** ordinary transactions — ledger/market inflows included — paused while a multi-block migration is incomplete: this FRAME multi-block-migration lockdown **is** item (1)'s inflow-freeze mechanism (verified on stable2606, SQ-281), so no state predicated on the half-migrated layout is created, and the runtime's automatic `MigrationHalt` halt-source bridge additionally freezes the execution queue. (A *source-less* staged halt — the SQ-274 genesis drill path with no live cursor — has no half-migrated layout to protect, so it gates the execution queue only; `playbook_calls(Migration)` constructs no separate guardian-dispatchable ledger/market freeze, SQ-280.) No partial-migration state is ever exposed to user calls.

**2. Pre-migration anchor (rewritten; SQ-127 + SQ-144 joint resolution, 2026-07-20).** PB-MIGRATION's audit anchor is the **last block before the new code's migrations could step**, recorded as `pre_migration_anchor = (block_number, block_hash)` in a **dedicated guard storage item**. Three normative changes from the superseded "state-root checkpoint at `execute()`" formulation, each forced by verification against the pinned stack.

- **Capture at application, not authorization.** The anchor is written by the guard's `on_initialize` on the **first block after the relay applies the new `:code`** — the block in which `System::parent_hash()` is the header hash of the last pre-migration block and no migration has yet stepped (`frame-executive` runs `on_runtime_upgrade`, then `frame_system::initialize`, then `on_initialize`, and only then the first migration step). Capturing at `execute()` cannot work: `DescriptorLeadTime` puts ≥ 72 h **plus unbounded keeper delay** between authorization and application (§2.2, §2.1(4)), so the superseded claim that "the divergence set … is exactly the migration's writes" was false by construction — it was all ordinary chain activity over ≥ 72 h *plus* the migration's partial writes. It is true only of an application-time anchor, and that is the whole reason the anchor exists.
- **No state root.** The anchor stores the block hash, not a storage root. The parent header already commits to the parent's `state_root`, so a separate root is redundant *given a correct hash* — and computing one on chain costs a full storage-root pass over the entire block's changed-key set inside `validate_block`, an unbounded PoV cost that no benchmark can account (the key tracker does not observe `storage_root`). An implementation MUST NOT call the storage-root host function to build this anchor. The hash bytes MUST be stored rather than looked up later: `BlockHashCount` prunes at 256 blocks (~25 min), far inside the 72 h retry-or-rollback window of item 3.
- **One home, and it is not the queue row or the event.** The anchor lives in its own storage item, for the lifetime defined below. It is **not** a field of `QueuedExecution` — §1.2(13) deletes that row on successful execute, i.e. before any migration can fail — and it is **not** carried on `UpgradeAuthorized`, whose shape is frozen at two fields by [02 §6](./02-integration-contract.md) and cannot gain one without a contract bump and joint sign-off. The anchor itself needs neither: it is guard-internal state read by operators and try-state, absent from the [02](./02-integration-contract.md) surface.

**Lifetime.** Two storage items, with distinct lifetimes. A **capture latch** is set where the go-ahead callback previously killed the checkpoint, and is consumed by the next `on_initialize`, which writes the anchor and clears the latch. The **anchor** is written after `PendingUpgrade` has already been cleared (that clear happens in the go-ahead block, one block earlier) and is itself cleared when the migration completes, when a recovery image applies, **or, if the applied image registered no migrations at all, never written in the first place — the latch is consumed without writing** — production wires `type Migrations = ()`, so no cursor is created and no completion callback ever fires; without this third clause the anchor would leak on every ordinary upgrade. The anchor's `block_number` is the number of the block its `block_hash` identifies (the last pre-migration block, i.e. `now − 1` at capture), never the capturing block.

The guard's try-state assertion runs in **one direction only**: *anchor present ⇒ (capture latch set ∨ a `MigrationHalt` source set ∨ a live migration cursor)*. It deliberately imposes **no** obligation in the `PendingUpgrade` direction. `PendingUpgrade` is written at `execute()` and the anchor at application, and `DescriptorLeadTime` puts ≥ 72 h plus unbounded keeper delay between them — so "pending upgrade with no anchor" is the *normal* state of a healthy chain for at least 43,200 blocks, and any assertion in that direction would fail continuously. The superseded both-Some/both-None **equality** is what made the old vacuity invisible: both items were killed together at the go-ahead callback, one block before any migration could step, so the assertion passed precisely because the anchor 09 depends on did not exist.

**3. Retry-or-rollback decision rule.**
- **Retry** (up to 2 attempts) iff the failure class is resource-bounded — weight/PoV overrun or a transient host error — using `pallet-migrations`' bounded continuation controls, which on stable2606 are **Root-only** (`force_set_active_cursor`/`clear_historic`/…), filtered to the D-13 "nobody" row (verified, SQ-274): available to bootstrap sudo in Phases 0–3, and post-sudo attainable only by riding the expedited-CODE lane like rollback — no guardian-dispatchable in-framework retry exists. A retry that completes clears the halt; `try-state` must pass before the freeze lifts.
- **Rollback** otherwise (logic fault, storage-shape mismatch, second failed retry): rollback is a **forward upgrade** carrying a remediation artifact whose migration restores the pre-migration-anchor semantics for the affected keys. It rides the **expedited CODE lane** (§3.1, admissible because the halt is active) with full attestation and fast-track ratification. Guardians cannot install code (kernel prohibition, carried); their playbook power ends at freezing (the automatic `MigrationHalt` bridge) and *initiating* the expedited-CODE lane — the in-framework retry controls are Root-only, not guardian-dispatchable (SQ-274).
- The decision MUST be taken within 72h of activation; inaction defaults to rollback initiation (status-quo-default discipline).

**4. Guardian notification and review.** Guardian activation and every playbook step emit `GuardianAction { power, target, justification_hash }`. The `MigrationHalted { cursor, failed_step }` diagnostic is emitted **once, on the first machine-trigger activation of a migration halt source** (a failed step, the stall predicate, an applied-code mismatch, or failed abort cleanup) — carrying the SDK cursor's exact bytes and reported failed step, and **not** re-emitted on a subsequent source or guardian step (normative; SQ-132(d) resolution, reconciled 2026-07-22). This binds it to the machine trigger, consistent with the paging rule below, rather than to the guardian lifecycle; it is a pallet-local operational diagnostic outside the frozen [02 §6](./02-integration-contract.md) ingest set by that section's (a)–(c) rule (the `PendingOutflowSyncFailed` class), so it carries no `INTEGRATION_CONTRACT_VERSION` change. The RB-UPGRADE runbook pages operators immediately — and paging begins at the **machine trigger above, not at activation** (clarified 2026-07-20): a stall satisfying that trigger already holds ordinary transactions under the multi-block-migration lockdown of item 1, so deferring the page until a guardian 5-of-7 activation would spend the outage on procedure. [12](12-release-and-operations.md) §6.3's Upgrades row carries the page-immediately marker for exactly this reason, making the two documents' paging obligations one rule rather than two. A mandatory retrospective `ratify` review is auto-scheduled; an unratified activation slashes 50% of approving members' bonds per the [06](./06-governance-and-guardians.md) rule.

## 4. Coretime continuity (D-9, resolves B-16)

The B-16 wedge — the dead-man freeze suppressing the TREASURY pipeline that funds the coretime renewal that would end the degradation — is closed by one enumerated, narrowly-scoped call:

`futarchy_treasury.execute_coretime_renewal(period_index)` — permissionless (Signed), keeper-rebated. Semantics:

- Spends **only** from the pre-authorized `ops.coretime` budget line in `pallet-futarchy-treasury` ([08](./08-treasury-and-economics.md)); before TREASURY arming (normally Phases 0–4) that line is provisioned through genesis/onboarding, and thereafter (normally Phase ≥ 5) it is replenished by ordinary TREASURY-class decisions. The stored ops multisig authenticates quotes but has no Signed authority to fund `ops.coretime`; its sole temporary funding exception is [07](./07-oracle-and-disputes.md) §8's runway-capped `OpsReserveProbe` top-up. The standing coretime authorization is therefore funded in calm weather, not during degradation.
- Dispatches the XCM DOT funding transfer from the treasury to the chain's renewal account on the Coretime chain, per the §6 XCM rules (renewal mechanics verified 2026-07-16, PLAN V-3: bulk renewal is `broker.renew` on the Coretime chain, Signed, price-capped, paid from the chain's funded renewal account there — this call funds exactly that account, no `Transact`). Route shape (PLAN V-19): the Coretime chain accepts DOT **only as a relay teleport** (`IsReserve = ()`), so the transfer is two-legged — a local reserve-withdraw of DOT to the relay, then a relay-side `InitiateTeleport` to the Coretime chain; the teleport executes on the relay and creates no teleport trust in this chain's executor (§6.1) **[VERIFY channel procedure at Phase-2/3 onboarding]**; idempotency key = `period_index`, bounded retry, keeper-monitored — an XCM failure never enters any decision or settlement path (I-24).
- **Exempt from every protocol-level freeze**: dispatchable during the dead-man freeze, hard-gate suspension, PB-LEDGER-FREEZE, and while the [08](./08-treasury-and-economics.md) §1.2 reserve-health flag is set (renewal is maintenance during exactly the degradations the flag accompanies — the D-9 wedge closure would be defeated by the "no new outflows" rule, so this line-debit is an explicit carve-out). **Scope note (amended 2026-07-18, B12):** during a PB-MIGRATION halt the FRAME multi-block-migration lockdown rejects *every* transaction while a migration cursor exists — no extrinsic, this one included, can bypass it without a custom executive; the renewal wedge there is accepted as bounded by the PB-MIGRATION runbook's halt/retire-and-rollback discipline ([06](./06-governance-and-guardians.md) §6.2; RB-UPGRADE), and the exemption list above names the freezes the runtime actually controls. It does not pass through the execution queue (§1.2(10) does not apply — it is not a queued proposal), cannot be blocked by guardian holds, and is the only **proposal-call** treasury outflow with this property. The separate runtime-internal `ops.reserve_probe` recovery envelope of [07](./07-oracle-and-disputes.md) §8 is the other narrow reserve-haircut carve-out; it is not an arbitrary transfer and can fund only the fixed probe program.
- Engaging it during degraded mode **does not consume a recovery epoch**: renewal is maintenance, not a governance decision, and the proposal-free recovery-epoch rule keys only on dead-man engagement itself, never on this call.
- Bounds: per-period spend ≤ the budget-line balance; the line's size is a [13](./13-parameters.md) value; the call is a no-op (error, no state change) when the renewal window is not open or the period is already funded.

**Quote authority, freshness and denomination (SQ-245/SQ-246 ruling, 2026-07-18).**

- **Quote source (authenticated).** The renewal quote `(period_index, price)` is supplied by the **operations multisig** (the [08](./08-treasury-and-economics.md) §2.1 ecosystem/ops administrator) via `futarchy_treasury.note_coretime_quote(period_index, price)` — Signed, accepted only from the stored **coretime quote authority** account (genesis-set to the ops multisig; rotatable, together with the renewal account below, by the `FutarchyTreasury` origin via `set_coretime_authority`). The operator reads the price from Coretime-chain sale state (the `broker` renewal price for the chain's core); authentication is the account check, and the blast radius of a wrong quote is bounded by the `ops.coretime` line balance plus per-period idempotency (a period funds at most once). Quote noting and expired-quote pruning are **freeze-exempt exactly like the renewal call itself** — a freeze that starved the renewal of a fresh quote would re-open the B-16 wedge.
- **Freshness and supersession.** A quote is executable while its age ≤ `ops.coretime_quote_ttl` blocks ([13](./13-parameters.md)); an expired quote cannot fund a renewal (error, no state change). Re-noting an open period **supersedes** the quote in place (price and noted-at reset; the ≤ 8 bound counts periods, not revisions). A quote leaves the bounded set on (i) supersession, (ii) successful funding, or (iii) `prune_coretime_quote(period_index)` — permissionless (Signed, keeper-cranked) **only once the quote is expired**; expiry is the sole permissionless prune trigger, and the quote authority may prune any open quote it noted.
- **Denomination.** The quote `price` is **DOT-denominated (planck)** — it is what `broker.renew` charges. `execute_coretime_renewal` debits `ops.coretime` by the **USDC value of the full DOT outflow** (`price` + the `ops.coretime_fee_dot` fee budget) converted at `ops.coretime_dot_rate` (µUSDC per DOT, [13](./13-parameters.md)), **rounded up** (against the spender); the call fails closed (no state change) when the line balance is below the converted debit or either key is unreadable. The rate is budget-envelope accounting only: DOT holdings are marked 0 in NAV ([08](./08-treasury-and-economics.md) §1.2), so no NAV or solvency term depends on it. The DOT itself is withdrawn from the treasury custody account's `ForeignAssets` DOT holding by the §6.1 route above.
- **Renewal account.** The renewal account on the Coretime chain is an **ops-operated account** (it must be able to sign `broker.renew` there — the verified route funds it, no `Transact`), stored beside the quote authority and rotated by the same `FutarchyTreasury`-gated call **[VERIFY at Phase-2/3 onboarding: account ceremony + HRMP channel procedure]**.
- **Genesis seating is an onboarding input (normative; added 2026-07-20).** Both accounts are *outputs* of that ceremony, so neither can be a chain-spec constant fixed in advance: until the ceremony runs there is no correct value to seat. The runtime therefore treats both as **unset-fails-closed** — with no stored quote authority no account can note a quote, and with no stored renewal account no renewal dispatches — which makes omission safe but also **silent**, since a genesis that simply forgets them looks identical to one awaiting the ceremony. Two obligations follow. The production genesis template and its chain-spec validator MUST carry both fields as explicit unfilled seats, so an operator preparing a production spec is confronted with them rather than defaulting past them; and seating them is a **Phase-3 entry gate** ([12](12-release-and-operations.md) §6.5), verified by noting and pruning one quote end-to-end. Development and test presets seat stand-ins and are exempt.

## 5. Phase-3 insider containment (D-13, resolves X-9)

### 5.1 Genesis-filtered calls — all origins, including sudo

The following `frame_system` calls are filtered **from genesis**, for **all** origins including the sudo key (the source's "post-bootstrap" qualifier is deleted):

```
system.set_storage
system.kill_storage
system.kill_prefix
system.set_code_without_checks
system.authorize_upgrade_without_checks
```

(`system.set_code` remains filtered as before; `authorize_upgrade`/`apply_authorized_upgrade` are the sole upgrade path, §2.) Enforcement: the SafetyFilter denies these unconditionally, **recurses through `sudo.sudo` / `sudo.sudo_unchecked_weight`** during Phases 0–3 so the multisig cannot reach them through the sudo wrapper, and **denies `sudo.sudo_as` outright** — it dispatches as caller-chosen `Signed(who)`, so recursing its inner call cannot contain it, and Root-dispatching `sudo` already covers the whole legitimate bootstrap surface (SQ-99, [06](./06-governance-and-guardians.md) §3.3); [15](./15-invariants-and-testing.md) carries negative tests for every wrapper × every filtered call. Consequence: even a fully malicious founding multisig cannot silently rewrite storage or install unchecked code — its worst case is bounded to the sudo-admissible surface, which is what the [14](./14-threat-model.md) founding-multisig threat row (new) prices.

### 5.2 Phase-3 real-USDC exposure caps

While real USDC trades under bootstrap authority, exposure is capped by two constitution keys (values in [13](./13-parameters.md)):

- `phase3.tvl_cap` — global cap on **total local USDC issuance** (all `ForeignAssets` USDC minted on this chain — a strict superset of, hence at least as tight as, the previous "ledger + treasury sovereign accounts" base; amended 2026-07-17, SQ-111: under the holdings base a deposit crediting a *user* account could never breach the cap at the inflow leg, which would have made the inflow-leg enforcement below vacuous); enforced at every `split`/inflow-crediting path and at the XCM inflow leg's **mint step**. The mint step is the *binding* enforcement point (only minting moves issuance); the `split`-path check is defense in depth under this base — a pure read (`total issuance ≤ cap`) that halts new escrow only while the chain is already over cap, since `split` moves already-issued USDC without changing issuance (ledger side: SQ-105/A2). **Split-path scope (normative; SQ-252 resolution, 2026-07-20):** the defense-in-depth read binds the **Signed** ledger split extrinsics only (`split`, `split_scalar`, `split_gate`, `split_baseline`). `MarketAuthority` splits on the D-3 trade path are exempt — they escrow collateral the trader already holds, every unit of which was metered at its own mint step, so they move no issuance and can create no unbacked claim; the cost of the exemption is that `buy`/`sell` escrow may still grow while the chain sits over cap. **Mint-step scope (normative; SQ-253 resolution, 2026-07-20):** the mint-step gate binds **issuance-increasing** mints. `pallet-xcm`'s trapped-imbalance reconstruction (§6.1) is exempt: the trapped holding is *already* counted in total issuance, the reconstruction mints a duplicate and immediately resolves it (net zero), and applying a prospective cap there would consume the trap entry while yielding no holding — destroying recoverable user funds rather than protecting the cap (R-7). The beneficiary deposit leg of any recovery remains fully metered by both keys. At the mint step an over-cap inbound reserve transfer fails on its first local instruction with nothing minted and nothing trapped locally ("fails politely"); the sender's USDC then sits in this chain's Asset Hub sovereign account pending recovery via the §6.1 trapped-assets/recovery row and the FE pre-check ([11](./11-frontend-workflows.md)) — the SQ-110 accepted reading.
- `phase3.deposit_cap` — per-account cumulative USDC deposit cap over the phase; enforced at the same points for **non-protocol beneficiaries**. Canonical protocol accounts ([03](03-conditional-ledger.md) §3/§4) are exempt from this per-account meter because treasury/trap-recovery and other protocol custody flows must not become permanently wedged behind a user-exposure ceiling; they remain fully subject to `phase3.tvl_cap`, and no protocol inflow creates or extends a `CumulativeDeposits` row. Retained protocol-account rows from an earlier runtime are inert and ignored by `try-state`. **Both caps MUST be enforced before any local mint where they apply (normative; SQ-129/SQ-251 resolutions, 2026-07-20/2026-07-23).** A cap refusal MUST fail the inbound message on its first local instruction with nothing minted and nothing trapped locally — the `tvl_cap` behaviour below, now required of `deposit_cap` too. The per-account check therefore binds a non-protocol beneficiary **extracted from the inbound program** (`DepositAsset{beneficiary}`) at or before the mint step: the beneficiary is unknowable at that point only through the `AssetTransactor` *interface*, never through the *message*, which carries it. The superseded reading — that a per-account rejection traps the already-minted holding, "recoverable by the sender via the §6.1 self-scoped `claim_assets` row" (SQ-110) — is **deleted as unachievable**. An inbound message's trap is keyed under the *sending* chain's origin, so the beneficiary can never self-claim it; and re-keying the trap to a non-protocol beneficiary does not rescue the design either, because a recovery claim's deposit leg stays metered (§6.1) by the same cumulative per-account cap the beneficiary has already exceeded — and that cap never clears within the phase, so the holding would re-trap on every attempt, bouncing forever. Trapping on a cap refusal converts a recoverable upstream failure into a permanently stranded one; G-1 requires the refusal to restore the status quo instead, and R-7 forbids the reading that strands a claim. The cumulative per-account meter is on-chain state shared by every enforcement point (`pallet-inflow-caps`; the XCM leg records through it transactionally, so an aborted message records nothing).

Both keys are **raised only by phase gates** (§7): they are not PARAM/META-adjustable during Phases ≤ 3; each phase-advancement upgrade carries the scheduled raise, and Phase 5+ sets them to unbounded sentinels. The FE surfaces the caps and the persistent **"bootstrap governance (sudo active)"** banner chain-read from `PhaseFlags` ([02](./02-integration-contract.md) chain identity; FE side [11](./11-frontend-workflows.md)) — sudo-era state is never presented as trust-equivalent to post-sudo state.

Every sudo use MUST be announced with a justification hash. The founding multisig is a first-class adversary in [14](./14-threat-model.md). Sunset: the removal migration of §7.2.

### 6.4 XCM health and reserve health

- **X sub-metric (XCM health)**: send/fail/timeout counters enter `C_onchain` ([05](./05-welfare-and-decision-engine.md)/[07](./07-oracle-and-disputes.md) own the pillar split). *(Resolved 2026-07-17, SQ-113 — formerly `[VERIFY XCM-health data availability on stable2603]`.)* On stable2606 (re-checked at the D-19 line move; XCM v5 unchanged) exactly three signals are admitted to `X`, all **local**: (1) a message accepted and delivered to the local transport by the router, (2) a local validate/deliver failure, and (3) the [07](./07-oracle-and-disputes.md) §8 reserve-probe timeout fold (the only locally-observable "sent but never answered" signal). v1 deliberately omits per-message response tracking for ordinary user and maintenance XCM, so their remote delivery/execution outcomes never enter a decision path by inferred success. - **R probe (reserve health)**: the sole authenticated class-3 XCM result admitted to `C_onchain`. A failing R sets the daily **C** breach flag toward status quo (R and X pillar placement owned by [05](./05-welfare-and-decision-engine.md) §4.3). If local XCM-health data is unavailable or ambiguous, R remains the pessimistic reserve-channel signal rather than a silent full-backing claim. ### 7.2 Sudo removal (Phase 3→4; carried, normative)

The Phase-3→4 runtime upgrade (a) removes `pallet-sudo` from the runtime, (b) runs a migration **asserting the sudo key storage is purged** and that no origin maps to Root outside the guard's internal path — the migration fails the upgrade if the assertion fails, (c) flips the execution guard from shadow to binding for PARAM, (d) applies the scheduled exposure-cap raise. It additionally refuses unless the bounded reserve-line bootstrap authority has already handed over: a successful positive `FutarchyTreasury` funding of `OpsReserveProbe` has occurred and `BootstrapOpsFundingClosed` is true. TREASURY arming by itself is not evidence of that refill path, and an armed-but-open latch is valid only during this bounded handover window. From the Phase-4 block, the authority matrix is complete and closed.

### 7.3 Preimage discipline (resolves B-13, pinning side)

At **qualification** (screening pass, slot won), `pallet-epoch` dispatches `preimage.request_preimage(payload_hash)`, pinning the committed payload for the proposal's whole lifecycle; the pin is released (`unrequest_preimage`) at every terminal state (`Executed`, `Rejected(*)`, expiry, withdrawal). Consequences: the unnote-after-adoption sabotage is dead (a requested preimage cannot be unnoted out from under the queue; any keeper can re-note the public bytes in the residual race), and the preimage-missing cancellation path now costs its proposer the 10% bond slash owned by [06](./06-governance-and-guardians.md)/[08](./08-treasury-and-economics.md). Playbook preimages are pinned the same way at playbook registration.
