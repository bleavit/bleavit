# Security Policy

## Reporting a Vulnerability

Please do not report security vulnerabilities through public GitHub issues, discussions, or pull requests.

Use GitHub's private vulnerability reporting feature instead:

1. Open the **Security** tab of this repository.
2. Select **Report a vulnerability**.
3. Describe the vulnerability and its potential impact.
4. Include reproduction steps, affected files, and a proposed fix when available.

Please provide enough information for us to reproduce and evaluate the issue.

## What to Include

A useful report should include:

- A clear description of the vulnerability
- The affected component or file
- Steps to reproduce the issue
- The expected and actual behavior
- The possible security impact
- Relevant logs, screenshots, or proof-of-concept code
- Any suggested mitigation or fix

Do not include secrets, private keys, access tokens, personal data, or other sensitive information that is not necessary for evaluating the report.

## Supported Versions

Bleavit is currently under active development.

Security fixes are applied to the current `main` branch and, when applicable, to the most recent supported release.

Older commits, development branches, forks, and modified deployments may not receive security updates.

## Response Process

After receiving a report, we aim to:

- Acknowledge the report within 3 business days
- Evaluate its severity and reproducibility
- Keep the reporter informed about meaningful progress
- Develop and test an appropriate fix
- Coordinate disclosure when necessary

Response times may vary depending on the complexity and severity of the vulnerability.

## Responsible Disclosure

Please give the maintainers reasonable time to investigate and fix the vulnerability before publishing details.

Do not exploit the vulnerability, access data belonging to others, disrupt services, or perform destructive testing.

We appreciate responsible security research that helps improve Bleavit.
